# Porting ENS to QRL Zond: a technical plan for QNS

**Bottom line up front.** QNS is buildable today on QRL Zond Testnet V2 with a **contract layer that is byte-for-byte ENS** plus a **JS SDK that swaps ecrecover-based signed records for ML-DSA-87 via `@theqrl/mldsa87`**. No prior QNS implementation exists on QRL as of April 2026 — this is greenfield. The hard problem is **not the contracts** (Hyperion accepts ENS Solidity unchanged and keccak256 works inside the EVM), it is **address representation**: QRL Zond addresses are 24 bytes with a separated 3-byte cryptographic descriptor, while ENS's canonical `IAddrResolver.addr(bytes32) returns (address)` hard-codes the 20-byte Solidity `address` type. The recommended resolution is to **treat EIP-2304/ENSIP-9 `addr(bytes32,uint256) returns (bytes)` as the primary forward-resolution record** and demote the legacy 20-byte `addr()` to a compatibility shim. Everything else — namehash, registry, reverse namespace, resolver pattern — ports cleanly. Below is the implementation-ready plan.

## 1. Executive summary

QNS ports **ENS v1 (not v2)** to QRL Zond. ENS Labs cancelled the Namechain L2 in February 2026 and now deploys ENSv2 only on Ethereum mainnet with an unfinalized per-name sub-registry model; porting v2 is a moving target. ENS v1 is frozen, battle-tested, and its contracts compile and run on **Hyperion** (QRL's Solidity fork for the QRVM/ZVM) with no source modifications required for the registry, resolver, and reverse-registrar layers.

The port splits into four layers: **(a) contracts** — vendored copy of `ensdomains/ens-contracts` at a pinned commit, with a single custom `QRLAddressResolver` profile replacing address assumptions; **(b) JS SDK** — fork of `ensjs` targeting `@theqrl/zond-wallet-provider` and `@theqrl/qrl_providers` as the injected wallet interface, with signed-record flows reimplemented against `cryptoSignSignature` from `@theqrl/mldsa87`; **(c) crypto layer** — ML-DSA-87 for any off-chain signed resolver records (EIP-5559-style writes, CCIP-Read gateway auth), keccak256 retained everywhere inside the EVM; **(d) naming policy** — FIFS registrar on a `.qrl` TLD for Phase 1, with a commit/reveal pricing controller deferred to Phase 3.

**The key insight driving the design**: QRL's Zond execution layer uses keccak256 internally for all EVM operations (storage slots, event topics, `CREATE2`, ABI hashing), so ENS's namehash algorithm works bit-identically on Zond. Post-quantum hashing (SHA2-256/SHAKE) only matters at the **public-key-to-address** boundary, which is a wallet-layer concern that ENS/QNS never touches directly. This is why the port is mostly mechanical.

## 2. Architecture overview

QNS has three stacks that mirror the ENS monorepo split (`ens-contracts`, `ensjs`, `ens-app-v3`):

The **contract stack** (`qns-contracts`) is a minimal fork of `ensdomains/ens-contracts` pinned to a post-ENSIP-19, pre-ENSv2 commit (roughly Q4 2025). It contains the ENSRegistry, Root, PublicResolver, ReverseRegistrar, UniversalResolver, and a new `FIFSQRLRegistrar` for the TLD. A single new resolver profile — `IQRLAddrResolver` — is added alongside the existing `IAddressResolver` to return native 24-byte QRL addresses as typed `bytes24` for ergonomics. NameWrapper, BaseRegistrarImplementation, ETHRegistrarController, and DNSSEC tooling are **excluded** from the initial port. Under Zond's planned ERC→QRC rebrand, the NFT-based registrar (if ever added) would implement QRC-721 rather than ERC-721, but this is Phase 3+ and should not block MVP.

The **SDK stack** (`qnsjs`) is a TypeScript library offering `resolveName(name)`, `lookupAddress(addr)`, `setAddr`, `setText`, and `setContenthash` against a Zond JSON-RPC provider. It imports `@theqrl/mldsa87` for any signature operations and uses `@theqrl/zond-wallet-provider` (EIP-6963, `window.qrl` injection per `@theqrl/qrl_providers`) for browser wallet connectivity. Because `@theqrl/web3` was archived in October 2023 and `zondjs` is still in development for Vortex IDE, QNS must target the wallet-provider layer directly rather than expecting an ethers.js-equivalent abstraction. The SDK exposes a UniversalResolver-style `resolve()` that makes a single RPC call.

The **crypto layer** is simply `@theqrl/mldsa87@1.1.1` (code-frozen Feb 13 2026, Halborn-audited March 2026 with all 13 informational findings resolved). QNS uses the `"ZOND"` default context string — a domain-separation value baked into the library — for all resolver record signatures, optionally extended to `"ZOND/QNS/v1"` to prevent cross-protocol replay. Keccak256 is provided by `@noble/hashes` (already a transitive dependency) and is used for namehash client-side, matching what the EVM does on-chain.

## 3. Minimum viable contracts and deployment order

The dependency-ranked deploy sequence is tight. **ENSRegistry** deploys first with no constructor args; it assigns root node `0x00...00` to the deployer. **Root** deploys next taking the registry address, and `ENSRegistry.setOwner(bytes32(0), rootAddr)` transfers root ownership. **PublicResolver** takes `(ens, nameWrapper=0, trustedController=0, trustedReverseRegistrar=reverseAddr)` — the zero addresses skip NameWrapper and controller pre-authorization, which is the correct posture for MVP. **ReverseRegistrar** takes only `(ens)` and is wired by calling `Root.setSubnodeOwner(labelhash("reverse"), deployer)` followed by `ENSRegistry.setSubnodeOwner(namehash("reverse"), labelhash("addr"), reverseRegistrarAddr)`, giving the contract control of the `addr.reverse` node at the canonical hash `0x91d177...5a9e2`. **FIFSQRLRegistrar** owns the `.qrl` TLD via `Root.setSubnodeOwner(labelhash("qrl"), fifsAddr)` and exposes `register(bytes32 label, address owner)` for permissionless first-come-first-serve allocation during the testnet phase. Finally, **UniversalResolver** deploys with `(owner, ens, batchGatewayProvider)` and registers its own reverse record via the `ReverseClaimer` helper.

The **critical omission** compared to mainnet ENS is the `.eth` auction stack: `BaseRegistrarImplementation`, `ETHRegistrarController`, `StablePriceOracle`, `DummyOracle`, and their price-feed wiring are **not** deployed. A permissioned or auction-based controller replaces FIFS in Phase 3 once economic policy is decided. NameWrapper is deferred indefinitely — the fuses/permissions system it adds is orthogonal to post-quantum naming and substantially increases attack surface.

**Contract-level changes required**: zero source edits to ENSRegistry, Root, ReverseRegistrar, UniversalResolver, or the base Resolver interfaces. One addition: a `QRLAddrResolver` profile exposing `qrlAddr(bytes32) view returns (bytes)` returning the native 24-byte address with descriptor, plus setter `setQrlAddr(bytes32, bytes)`. One modification: PublicResolver's inherited `IAddrResolver.addr(bytes32)` is retained but should internally truncate/transform from the stored bytes representation — see §5 for the address compatibility analysis.

## 4. Cryptography integration

The ENS signing surface is narrower than it looks. **Inside the EVM nothing changes**: keccak256 remains the hash for namehash, `CREATE2`, storage slots, event topics, and ABI encoding, and Zond's Hyperion compiler and QRVM execution layer preserve these semantics. There is no `ecrecover` in any ENS or QNS core registry/resolver write path; ownership is enforced by `msg.sender` checks via the transaction signature, which Zond handles at the consensus layer using ML-DSA-87.

**The real integration points are three off-chain signing flows**: (1) ENSIP-19 signature-based reverse name setting (`setNameForAddrWithSignature`, selector `0x2023a04c`), which on Ethereum is an EIP-712 signature verified via `ecrecover`; (2) EIP-5559 / ERC-7700 off-chain resolver writes, where a client signs record updates that a gateway relays; (3) CCIP-Read (EIP-3668) gateway responses when resolvers revert with `OffchainLookup` and the client fetches signed data from a URL. All three must be reimplemented against ML-DSA-87.

The pattern for each is identical: replace the `ecrecover`-based verifier with a Solidity verifier that calls an **ML-DSA precompile** if one exists on Zond (a precompile for ML-DSA-87 verification would follow the go-ethereum precompile pattern at a reserved address, and its existence and gas cost need direct verification in the `go-zond` source — see open questions §7), or falls back to a full in-EVM ML-DSA verifier (prohibitively expensive, ~millions of gas). On the client side, `@theqrl/mldsa87` exposes `cryptoSignSignature(sig, msg, sk, randomizedSigning=true, ctx="ZOND/QNS/v1")` for signing and `cryptoSignVerify(sig, msg, pk, ctx)` for verification; signatures are 4,627 bytes and public keys are 2,592 bytes, which **breaks any on-chain storage pattern that assumed 65-byte secp256k1 signatures**. For QNS, signed messages should carry the public key in the payload or reference it by namehash pointer rather than attempting to store signatures on-chain.

For Phase 1 forward resolution, **none of this matters** — ownership checks on the registry are done by the transaction sender, and Zond's ML-DSA tx signatures are verified at consensus. The crypto integration only becomes critical in Phase 2 (ENSIP-19 reverse with signatures) and Phase 4 (identity extensions, CCIP-Read).

## 5. Address and namehash compatibility analysis

**Namehash is a non-issue.** EIP-137 defines namehash recursively as `keccak256(namehash(parent) || keccak256(label))` with UTF-8 label bytes, normalized per ENSIP-15 (which supersedes UTS-46). Zond's QRVM runs keccak256 natively, so on-chain and off-chain namehashes agree. Use `@adraffy/ens-normalize` client-side (already the reference for viem and ensjs) before hashing labels. The namehash of `"qrl"` and `"addr.reverse"` computes identically on Zond.

**Address is the hard problem.** QRL Zond's current address format (as of Testnet V2, March 2026) is 24 bytes displayed with a "Q" prefix, and as of the April 2025 go-qrllib refactor the 3-byte cryptographic descriptor is stored as a **separate field**, not embedded in the address bytes. The final mainnet format is roadmapped to keep this arrangement with descriptor separated. This conflicts with three ENS assumptions: Solidity's `address` type is 20 bytes (EVM word slot with the high 12 bytes zero); `IAddrResolver.addr(bytes32) returns (address)` returns that 20-byte type; and `ReverseRegistrar.sha3HexAddress` does `keccak256(lowercase_hex(addr, 40 chars))` — hard-coded to 40 hex chars.

There are three viable paths. **Path A (preferred): dual-stack resolver.** Keep the stored primary record as bytes via a new `qrlAddr(bytes32) returns (bytes)` (type-tagged as the native 24-byte form with descriptor retrievable from Zond's account metadata). Implement the legacy `addr(bytes32) returns (address)` as a lossy compatibility shim that returns the last 20 bytes (or reverts if the consumer requires full precision). Use EIP-2304/ENSIP-9 `addr(bytes32, uint256 coinType) returns (bytes)` with a **new coinType** assigned to QRL Zond — per ENSIP-11's scheme, `coinType = 0x80000000 | chainId` produces a self-describing value once Zond's chainId is published. This path preserves tooling compatibility (ensjs-style libraries "just work" for the 20-byte shim) while giving QNS-native consumers the full address. **Path B: Hyperion address redefinition.** If Hyperion redefines the `address` type to 24 bytes at the language level (which would be a major breaking change from ENS-compatible Solidity — unverified, flagged as a top open question), all ENS contracts would need recompilation and possibly signature changes. This is a project risk, not a design choice. **Path C: strip descriptor, store 20 bytes.** Treat the descriptor as wallet metadata never surfaced in QNS, store only the 20-byte tail. Simplest, but loses information the ecosystem may later require and assumes the 20-byte tail is collision-resistant on its own.

**Recommendation**: adopt Path A. The ENSIP-19 reverse namespace computation (`sha3HexAddress` hashing the 40-char hex lowercase) should be replaced with a `sha3QRLAddress` that hashes the 48-char lowercase hex of the 24-byte address (no `Q` prefix, no `0x`, no descriptor). This keeps the reverse lookup deterministic and chain-specific. The per-chain reverse namespace (ENSIP-19 second form) `[coinTypeAsHex].reverse` gives QNS forward-compatibility with any ENS client that eventually speaks to QRL Zond via UniversalResolver.

## 6. Phased implementation roadmap

**Phase 1 — Basic forward resolution on Testnet V2 (target: 4-6 weeks).** Vendor `ens-contracts` at a pinned Q4 2025 commit. Deploy ENSRegistry, Root, FIFSQRLRegistrar (owning `.qrl`), and PublicResolver (with NameWrapper and controllers zeroed). Register a handful of test names via FIFS, set resolvers, and populate `qrlAddr` records. Build a minimal `qnsjs` with `resolveName(name) → bytes24` that walks registry → resolver and calls `qrlAddr(namehash)`. Validate against at least two names, two accounts, and round-trip through a dapp using `@theqrl/zond-wallet-provider`. Output: deployable contracts, library on npm as `@qns/core` and `@qns/sdk`, a one-page demo.

**Phase 2 — Reverse resolution and Universal Resolver (target: 3-4 weeks after Phase 1).** Deploy ReverseRegistrar and wire `addr.reverse` per ENSIP-19's per-chain form using the QRL coinType (`0x80000000 | zondChainId`). Deploy UniversalResolver configured with the QNS registry and a stub batch-gateway provider. Extend `qnsjs` with `lookupAddress(qrlAddr) → string` including the forward-confirm check that ENS mandates. Defer signature-based `setNameForAddrWithSignature` to Phase 4 — the basic `setName(string)` path works without ML-DSA signatures because it relies on the transaction sender. Output: reverse resolution end-to-end, single-RPC UniversalResolver lookups, primary-name support in the demo dapp.

**Phase 3 — Records, text, and contenthash (target: 2-3 weeks).** Enable the remaining PublicResolver profiles: `ITextResolver` for EIP-634 text records (avatar, email, url, com.twitter, etc.), `IContentHashResolver` for EIP-1577 contenthash, and `IAddressResolver` multichain addresses. Publish SLIP-44 / ENSIP-11 guidance for QRL Zond's coinType. Extend `qnsjs` with `getText`, `setText`, `getContenthash`, `setContenthash`. Output: feature parity with ENS for record retrieval, adequate for dapp avatar display and IPFS content routing.

**Phase 4 — Identity extensions and ML-DSA signed records (target: 6-8 weeks).** Implement ENSIP-19 signature-based reverse with ML-DSA-87 verification. This requires either a precompile on Zond (verify with go-zond maintainers whether one is allocated; if not, lobby for inclusion before mainnet — see §7) or an in-EVM Dilithium verifier (ballpark 5-10M gas, likely acceptable for infrequent reverse setups but painful). Build a CCIP-Read gateway for off-chain records signed with ML-DSA-87, adapting the EIP-3668 `OffchainLookup` flow. Add pubkey resolver profile for publishing ML-DSA public keys as part of on-chain identity — this is QNS's natural extension beyond what ENS offers, since ML-DSA pubkeys (2,592 bytes) are exactly the kind of artifact a post-quantum identity system should anchor to names. Output: signed record workflows, off-chain data gateway, post-quantum pubkey publication as a first-class QNS record type.

**Phase 5 — Mainnet and economics (timeline tied to Zond mainnet).** Replace FIFS with a commit/reveal registrar modeled on ETHRegistrarController, with pricing in QRL (or Planck/Shor sub-units per the roadmap rebrand). Decide on grace period, renewal policy, and referrer fees. Submit a QIP to the `theQRL/qips` repository formalizing QNS as an ecosystem standard — none exists today and the QIP process is active (custodians: jackalyst, fr1t2, jplomas). Coordinate with any eventual QRC-721 standard for tokenized name ownership.

## 7. Open questions requiring community input

The plan has **seven load-bearing unknowns** that need direct verification with QRL core developers (Discord, GitHub Discussions, or the QIP process) before committing to contract deployment. These should be resolved in a pre-implementation discovery spike of 1-2 weeks.

**First**, does Hyperion redefine the Solidity `address` type from 20 bytes to 24 bytes, or does it preserve the 20-byte type and surface the full 24-byte QRL address only via bytes/struct abstractions? This single answer determines whether ENS contracts compile unmodified or whether every `address` field in the registry needs retyping. **Second**, is there an ML-DSA-87 verification precompile allocated on Zond, and if so at what address and gas cost? The existence of such a precompile makes Phase 4 straightforward; its absence makes signed records expensive or forces off-chain verification via CCIP-Read. **Third**, what is Zond's chainId for Testnet V2 and planned mainnet? This is needed to derive the ENSIP-11 coinType and the per-chain ENSIP-19 reverse namespace. **Fourth**, does the on-chain stored representation of an account address include the 3-byte descriptor or exclude it (post-April-2025 descriptor separation)? The sha3QRLAddress reverse-node computation must match whatever on-chain `msg.sender` resolves to. **Fifth**, which TLD is politically and technically viable: `.qrl`, `.q`, `.zond`, or `.qns`? The roadmapped rebrand away from "Zond" nomenclature argues against `.zond`; `.qrl` aligns with the final brand; `.q` is shortest but risks collision with non-blockchain DNS efforts. Submit this as the first QIP under the QEP (Enhancement Program) track. **Sixth**, is `ecrecover` (precompile 0x01) retained on Zond for compatibility with ported Ethereum contracts, or is it removed since ECDSA has no place in a post-quantum chain? This affects every OpenZeppelin contract that uses `ECDSA.recover`, not just ENS. **Seventh**, what is the status of `zondjs` (the successor to the archived `@theqrl/web3`), and is it production-ready for dapp integration, or should QNS target the lower-level `@theqrl/qrl_providers` EIP-1193 interface directly?

Secondary design questions with lower blocking weight: whether to mirror the ENSv2 per-name sub-registry architecture (risky — spec not finalized) or stay on the v1 monolithic registry (recommended — stable, well-audited); whether to support XMSS keys as an alternative in pubkey records alongside ML-DSA (XMSS is deprecated for validators but still available for wallets); and whether QNS names should tokenize as QRC-721 from day one (recommended: no, defer until QRC-721 standard itself is finalized).

## Conclusion

QNS is a **mechanical port at the contract layer and a crypto-library swap at the SDK layer** — the ENS codebase is in a lucky architectural position where namehash uses only keccak256 (available on Zond), ownership uses only `msg.sender` (handled by Zond's ML-DSA tx signing at consensus), and the only Ethereum-specific assumption that bleeds into the design is the 20-byte `address` type, which EIP-2304 already provides a variable-length escape hatch for. The substantive engineering work sits in three places: **address representation** (resolve Path A vs. B vs. C by reading Hyperion's type system), **ML-DSA signed records** (contingent on precompile availability), and **coordination** (publish the first naming-related QIP, pick a TLD, engage core devs on the seven open questions). Phase 1 forward resolution is a 4-6 week milestone that needs only public information already gathered here plus two or three clarifying answers from the QRL core team. Phases 2-3 extend naturally from Phase 1. Phase 4 is where QNS moves beyond being "ENS on Zond" and becomes **a post-quantum identity primitive** — the point at which publishing ML-DSA-87 public keys as first-class name records enables dapp-signed actions, verifiable credentials, and gossip-network identity that have no counterpart in ENS. That, not basic name resolution, is the feature that makes QNS worth building rather than shipping with `addr.qrl.domains`-style off-chain aliases.
